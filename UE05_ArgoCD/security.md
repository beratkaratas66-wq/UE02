# GitOps & ArgoCD Security Analysis - Capstone Exercise

## Part A: ArgoCD Setup
**Why a dedicated namespace?**
Running ArgoCD in its own namespace (`argocd`) enforces isolation. ArgoCD requires highly privileged ServiceAccounts to deploy resources across the entire Kubernetes cluster. Isolating it prevents other applications (like the recipe-api) or compromised pods from accessing ArgoCD's secrets, configuration maps, and internal API.

**The default Admin Password:**
The default password is auto-generated by the ArgoCD API server upon installation and stored as a K8s Secret named `argocd-initial-admin-secret` in the `argocd` namespace. It must be changed immediately because anyone with `get secrets` permissions in the cluster could read this password and gain full GitOps-level control over production, effectively bypassing all git controls.

**App-of-Apps Pattern:**
The App-of-Apps pattern uses a single "root" Application resource that points to a Git directory containing other Application manifests. 
*Security Benefits:* It ensures that even the ArgoCD configuration itself is managed declaratively via Git. Creating a new application or modifying deployment targets requires a Pull Request, code review, and leaves a permanent audit trail, rather than an admin making untraceable changes manually via the UI.

## Part B: ArgoCD Security Hardening
**ArgoCD RBAC vs. Kubernetes RBAC:**
* *Kubernetes RBAC* controls access to the K8s API server.
* *ArgoCD RBAC* controls access to the ArgoCD API and UI.
Both are required because ArgoCD itself operates with a highly privileged K8s ServiceAccount. If a user has low K8s privileges but full ArgoCD UI privileges, they could bypass K8s RBAC by telling ArgoCD to deploy malicious resources on their behalf.

**Repository Access Security:**
* *Blast Radius:* If ArgoCD's repository credentials leak, an attacker can clone and read the source code.
* *Read-Only Deploy Keys vs. PAT:* A Personal Access Token (PAT) is tied to a user account and often carries extensive scopes (including push access). A read-only SSH Deploy Key is scoped tightly to a single repository and strictly prevents attackers from pushing malicious code or altering the Git truth, drastically reducing the impact of a compromised ArgoCD instance.
* *Storage:* ArgoCD stores repository credentials securely as Kubernetes Secrets in the `argocd` namespace.

**Sync Policy & Drift Detection:**
* *Drift:* Drift occurs when the live state of the Kubernetes cluster diverges from the declared state in Git (e.g., an admin runs a manual `kubectl edit deployment` to change an image tag).
* *selfHeal: true* tells ArgoCD to automatically overwrite any manual cluster changes back to the state defined in Git. This is a crucial security feature as it instantly remediates unauthorized runtime modifications or tampering.
* *prune: true* ensures that if a resource is deleted from Git, ArgoCD deletes it from the cluster. Security implication: It prevents "orphaned" resources (like outdated, vulnerable pods or forgotten exposed services) from lingering in the cluster and expanding the attack surface.

## Part C: GitOps Threat Model
### 1. Supply Chain Threats
**The Trust Chain:** Developer -> Git Repo -> ArgoCD -> Helm Chart -> Container Registry -> K8s
* **Developer:** Compromise = stolen credentials. Gain = ability to push malicious commits.
* **Git Repo:** Compromise = broken branch protections. Gain = manipulation of the "single source of truth".
* **ArgoCD:** Compromise = UI/API takeover. Gain = ability to deploy arbitrary K8s manifests bypassing Git.
* **Container Registry:** Compromise = registry access. Gain = replacing legitimate application images with malware (supply chain injection).

**Malicious Helm Chart Attack:** If an attacker pushes a malicious chart, ArgoCD will faithfully deploy it because ArgoCD blindly trusts Git. Detection requires continuous vulnerability scanning and strict PR reviews. Controls from Ex 2 (Image Signing via Cosign and SBOMs) mitigate this: an admission controller (like Kyverno) would reject the deployment at the K8s level because the malicious image signature would not match the trusted developer's key, breaking the deployment chain safely.

### 2. Access Control Analysis
* **ArgoCD ServiceAccount:** Has cluster-admin (or very broad) privileges to deploy resources. Necessary, but high risk.
* **Developers (Git Collaborators):** Have write access to Git. Least privilege dictates they should require PR approvals and not push directly to `main`.
* **K8s Admins:** Have direct `kubectl` access. In a pure GitOps environment, this breaks least privilege. Direct cluster access should be revoked or restricted to emergency "break-glass" roles.
* **CI Pipeline:** Has push access to the image registry. Should use scoped, short-lived OIDC tokens instead of static credentials.

### 3. Incident Response
* **Scenario A (Unknown Image in Cluster):** Because `selfHeal` is enabled, ArgoCD should automatically revert unauthorized K8s changes. If the image persists, the attacker either bypassed ArgoCD (e.g., compromised the K8s API directly) or disabled ArgoCD. Response: Isolate the node, cordon the pod, audit K8s API logs to find the deploying identity, and revoke compromised K8s credentials.
* **Scenario B (Compromised GitHub Account pushes malware):**
    ArgoCD will automatically deploy the malicious change. Response: 
    1. Immediately revoke the developer's Git access and SSH keys.
    2. Use `git revert` on the malicious commit and push to `main` (ArgoCD will instantly sync and roll back the cluster).
    3. Rotate any application secrets (database passwords, API keys) that the malicious code may have exfiltrated while running.
